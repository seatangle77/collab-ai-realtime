## 时间与时区统一约定（后端 & 前端）

### 1. 存储层（数据库）

- 所有时间字段统一使用 PostgreSQL 的 `timestamp with time zone` 类型。
- 插入时一律使用 **UTC 时间**（例如：
  - SQL 中使用 `NOW()`（建议数据库时区设为 UTC）；或
  - 在应用层生成 `datetime.now(timezone.utc)` 再传给 SQL）。
- 数据库中的时间只作为「绝对时间点」，**不做任何东八区转换**。

### 2. 后端 API 行为

- Pydantic 模型中所有时间字段（例如 `created_at`、`last_updated`、`ended_at`）统一使用 `datetime` 类型。
- 返回给前端的时间一律是 **UTC 的 ISO8601 字符串**，例如：
  - `2026-03-04T12:00:00Z`
  - 或 `2026-03-04T20:00:00+08:00`（等价于有 tzinfo 的 UTC 时间）
- 对于客户端传上来的时间字段：
  - 要求使用带时区信息的 ISO8601 字符串；
  - FastAPI / Pydantic 会自动解析为 `datetime`（内部视为 UTC 时间点）；
  - 后端不负责转换成东八区，只负责转成 UTC 后入库。

### 3. 前端约定

- 前端收到的所有时间都视为 **UTC 时间点**：
  - 展示给用户之前，统一转换为用户所在时区显示（当前产品约定为东八区 `Asia/Shanghai`）。
- 前端表单/时间选择器：
  - 用户操作的是东八区本地时间；
  - 在发送给后端之前，将本地时间转换为 **UTC 或带 `+08:00` 的 ISO 字符串**，再提交给 API。

### 4. 设计原则

- **数据库 & 后端内部统一使用 UTC**，保证所有服务（未来多机房、多时区）行为一致。
- **时区展示及本地化逻辑尽量放在前端**，后端只负责「时间点」本身和一致的格式，不关心具体显示成「北京时间」还是其他时区。
